# System Design Daily Deep-Dive Rules

## Purpose

Every learning day in this repository must be treated as a **deep analysis session**, not a quick overview.

The goal is to build interview-ready depth for a **Senior Platform Engineer / SRE / Kubernetes / Cloud / DevOps** profile.

---

## Mandatory Rule

For every day, do not stop at definitions or short notes.

Each topic must include:

1. Concept explanation
2. AWS implementation angle
3. Azure implementation angle
4. Kubernetes / platform engineering angle
5. Real-world production usage
6. Architecture diagram
7. Failure modes
8. Scaling strategy
9. Security controls
10. Observability requirements
11. Cost and operational trade-offs
12. Interview questions and answers
13. One practical design exercise
14. Final revised answer for revision notes

---

## Daily Study Structure

Use this structure for each session:

```text
10 min  - Revise previous topic
15 min  - Understand core concept
15 min  - Compare AWS, Azure, and Kubernetes implementation
15 min  - Work through a real-world design scenario
10 min  - Discuss failure, security, observability, and cost
10 min  - Write final interview-ready revision notes
```

The time can exceed 1 hour when the topic requires deeper analysis.

---

## Daily Output Required

At the end of every day, write the final output into:

```text
SYSTEM_DESIGN_REVISION_NOTES.md
```

Each day's notes must include:

```text
1. Topic name
2. Date
3. Key concepts learned
4. AWS version
5. Azure version
6. Kubernetes/platform version
7. Architecture diagram
8. Failure scenarios
9. Scaling approach
10. Security design
11. Observability design
12. Cost considerations
13. Interview Q&A
14. Final polished answer
15. Weak areas to revise
```

---

## Deep-Dive Review Checklist

Before marking any day complete, verify:

- Did we explain the concept clearly?
- Did we cover AWS and Azure equivalents?
- Did we connect the topic to Kubernetes/platform engineering?
- Did we include a production-style architecture diagram?
- Did we discuss failure scenarios?
- Did we discuss HA/DR where relevant?
- Did we include security controls?
- Did we include observability and alerting?
- Did we include cost/operational trade-offs?
- Did we convert the answer into interview-ready language?
- Did we update the revision notes file?

---

## Interview-Ready Answer Standard

Every final answer should follow this structure:

```text
1. Clarify requirements
2. State assumptions
3. Estimate scale
4. Explain high-level architecture
5. Explain component responsibilities
6. Explain data flow
7. Explain scaling
8. Explain reliability and failure handling
9. Explain security
10. Explain observability
11. Explain cost and trade-offs
12. Summarize final design
```
